# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview
ArcadeCore is a Paper/Spigot Minecraft plugin (group `org.drappula`, target: Paper 1.21.11) providing core lobby/arcade-server functionality: a game/match/queue framework, a SQLite-backed arena map pool, per-player profiles, and a SQLite-backed user data store. It is split into two Gradle subprojects:
- **ArcadeAPI** — a `java`-only module of public interfaces/events other plugins depend on to integrate with ArcadeCore (package `org.drappula.arcadeApi`). Contains no implementation.
- **ArcadePlugin** — the actual plugin (`java-library` + Kotlin), package `org.drappula.arcadeCore`, implements every `ArcadeAPI` interface and is what gets shaded into the runnable jar.

## Build & run commands
Run from the `ArcadeCore/` root (this is the Gradle root project; `ArcadeAPI`/`ArcadePlugin` are subprojects).
- `./gradlew build` — compiles both modules and produces the shadowed plugin jar (shadowJar runs as part of `build`).
- `./gradlew :ArcadePlugin:shadowJar` — build the relocated, dependency-bundled jar directly (output in `ArcadePlugin/build/libs`).
- `./gradlew :ArcadePlugin:runServer` — launches a local Paper 1.21.11 test server with the plugin installed (via `xyz.jpenilla.run-paper`); use this for manual in-game testing. Server heap is set to `-Xms2G -Xmx2G`.
- There is no test suite configured in this project.

## Architecture
- **Language/toolchain**: Java 21 (ArcadeAPI) and Java 21 + Kotlin (ArcadePlugin, `kotlin("jvm")` applied, JVM toolchain 21), built with Gradle Kotlin DSL.
- **Shadow/relocation**: `com.gradleup.shadow` (applied only in `ArcadePlugin`) bundles `dev.dejvokep:boosted-yaml` (core + spigot-serializer) and `org.xerial:sqlite-jdbc`. `boosted-yaml` and `sqlite-jdbc` are *also* declared as runtime libraries in `ArcadeCoreLoader` (via a `MavenLibraryResolver` from Maven Central), which resolves them at plugin load time before the classloader is set up — the shadow-bundled copies and the loader-resolved copies coexist by design; don't assume removing one is safe without checking the other.
- **Plugin entry point**: `ArcadeCore` (`ArcadePlugin/src/main/java/org/drappula/arcadeCore/ArcadeCore.java`) is a singleton accessible via `ArcadeCore.get()`. `onEnable()` runs, in order: `setupConfig()` (loads `DataConfig`, `MainConfig`, `MessagesConfig`), `connectDatabase()` (opens the SQLite connection, throws on failure), `MapManager.get().load()` (reads the map pool from the database), `registerCommands()` (Brigadier via `LifecycleEvents.COMMANDS` — no `plugin.yml`, only `paper-plugin.yml`), `registerListeners()`, `registerAPI()` (publishes `ArcadeAPIImpl` through `ArcadeAPIProvider`). `onDisable()` reloads `GameManager` (clears in-memory game/match/participant state), releases all maps (`MapManager.releaseAll()`, clearing `in_use` flags in memory and DB), and disconnects the database.
- **API/impl split**: Every cross-cutting system is defined as an interface in `ArcadeAPI` (`IGameManager`, `IMatchManager`, `IQueueManager`, `IMapManager`, `IMatch`, `IParticipant`, `IArcadeMap`) and implemented as a singleton (`X.get()` static accessor, lazily constructed) in `ArcadePlugin`'s `managers/` package (`GameManager`, `MatchManager`, `QueueManager`, `MapManager`, `Match`, `Participant`, `ArcadeMap`). `ArcadeAPIImpl` (`ArcadePlugin/.../api/ArcadeAPIImpl.java`) is the single class that bridges the two, registered once in `ArcadeCore#onEnable`. When adding a new cross-plugin-facing feature, add the interface/event to `ArcadeAPI` first, then wire the concrete manager and expose it through `ArcadeAPIImpl`.
- **Game/match/queue flow**: `GameManager` holds registered `Game`s plus in-memory `matches`/`participants` maps keyed by game ID (no persistence — cleared on `reload()`, called from `onDisable`). `QueueManager.joinQueue` fires a cancellable `QueueEnterEvent`; when a queue reaches `game.getMinPlayers()` a `QueueCountdownTask` starts (`queue.start-countdown` in `MainConfig`, default 20 s, ticks 1 s), which starts the match when it expires — or immediately once the queue reaches `game.getMaxPlayers()`. Dropping below the minimum (via `leaveQueue`) cancels the countdown. `startQueuedMatch` slices up to `getMaxPlayers()` players off the queue and calls `MatchManager.startMatch`, which can return **null** — if `game.createArena(players)` returns null it acquires a map from `MapManager` (none available → null, queued players get the `map-unavailable` message), then bails if the map has fewer spawn points than players, then fires a cancellable `MatchStartEvent` (cancelled → depopulate + release map + null). On success it teleports each player to a spawn point and starts `MatchStartTask` (repeating `BukkitRunnable`, countdown from `match.start-countdown` in `MainConfig`, ticks every 20 ticks/1s). Matches carry a `MatchState` (`LOADING → STARTING → STARTED → ENDING → ENDED`), advanced by `startMatch`/`MatchStartTask`/`endMatch`/`MatchEndTask`. `Match.end()` → `MatchManager.endMatch` fires `MatchEndEvent` and starts `MatchEndTask`, which eliminates all remaining participants immediately (via `MatchManager.eliminateParticipant`, which applies `GameEndSettings` gamemode/fly overrides or config defaults), then — 5 seconds later — depopulates the match, releases its map, and sends spectators back to the lobby. There's no separate "declare winner" step wired up yet; `getWinnerParticipants()`/`setWinnerParticipants()` exist on `Match` but nothing currently calls the setter.
- **Map pool**: `MapManager` (`managers/map/`) keeps an in-memory `List<ArcadeMap>` loaded from SQLite at enable via `MapDataManager` (`database/MapDataManager.java`, raw JDBC against the `maps` + `map_spawns` tables); maps whose world isn't loaded are skipped with a warning. `acquireMap(gameId)` returns the first map that is enabled, not in use, and has ≥1 spawn point, flagging it `in_use` both in memory and in the DB (so a crash doesn't leak maps forever — `releaseAll()` on disable clears the flags). All mutations (`createMap`, `addSpawn`, `setEnabled`, `deleteMap`) write through to the DB and update the in-memory list. Maps are administered entirely in-game via `/arcade map ...` — there is no file-based map config.
- **Config layer**: Config classes extend `Config` (`config/Config.java`), which wraps `dev.dejvokep.boostedyaml.YamlDocument` with `SpigotSerializer` (so Bukkit types like `Location` (de)serialize directly) and `BasicVersioning` keyed on a `version` field for auto-updating the file to the packaged default. Three configs, each with a static `setup()`/`get()`:
  - `DataConfig` (`data.yml`) — runtime/mutable state (currently just `SPAWN_LOCATION`, via `setSpawnLocation`/`getSpawnLocation`).
  - `MainConfig` (`config.yml`) — admin-facing settings, e.g. `lobby.teleport-spawn`, `lobby.gamemode`, `match.start-countdown`, `match.end.gamemode`, `match.end.fly-enabled`, `queue.start-countdown`.
  - `MessagesConfig` (`messages.yml`) — all player-facing text (e.g. `match-started`, `countdown-message-singular`/`-plural`, `participant-eliminated`, `match-ended`, `map-unavailable`, `map-created`, `map-not-found`, `bad-arguments.unknown-game`), sent via `MessageUtil.sendMessage` (MiniMessage + Adventure `Placeholder`/`TagResolver`).
- **Commands**: `MainCommand.get()` builds a single Brigadier tree under `/arcade`: bare `/arcade` prints plugin name/version/description; `reload` reloads `DataConfig` only (not `MainConfig`/`MessagesConfig`); `setspawn` saves the invoker's location; `start <game>` (requires `arcade.force-start` permission) force-starts a match via `QueueManager.forceStart`; `queue <game>` has tab-completion wired (`suggestGames`) but its handler is currently a no-op — it doesn't call `QueueManager.joinQueue`; `map create|addspawn|enable|disable|list|delete` (requires `arcade.map.admin` permission) administers the map pool — `create` uses the invoker's current world, `addspawn` the invoker's exact location.
- **Database**: `Database` (`database/Database.java`) opens a single static SQLite connection to `<plugin-data-folder>/database.db` on enable (with `PRAGMA foreign_keys = ON`) and creates five tables if absent: `user_profiles` (uuid, username), `games` (game_id), `game_stats` (uuid, game_id, points, wins, losses, other_stats JSON — FKs cascade-delete), `maps` (map_id, game_id, display_name, world, enabled, in_use), and `map_spawns` (map_id + spawn_index PK, x/y/z/yaw/pitch — cascade-deletes with its map). `user_profiles` is read/written via `UserDataManager` (`get`/`getOrCreate`/`save`, upsert on conflict), `maps`/`map_spawns` via `MapDataManager` — `games`/`game_stats` are provisioned but unused so far.
- **Listeners**: `LobbyListener` handles `PlayerJoinEvent` — registers a `Profile`, teleports to `DataConfig`'s saved spawn (if `lobby.teleport-spawn` is enabled), resets health to max, sets gamemode from `MainConfig` — and `PlayerQuitEvent` — leaves any queue, eliminates the player from any match they're still alive in, removes them from spectator lists, and removes the `Profile`.
- **Profiles**: `ProfileManager` is a simple in-memory `UUID -> Profile` map (register/get/remove) separate from the persisted `UserData`/`user_profiles` table — it's runtime session state, not database-backed.
- **Resource filtering**: `paper-plugin.yml` uses Gradle `processResources` token expansion for `${description}`/`${version}`, sourced from `gradle.properties`.
