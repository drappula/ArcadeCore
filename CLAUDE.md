This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview
ArcadeCore is a Paper/Spigot Minecraft plugin (group `org.drappula`, target: Paper 1.21.11) providing core lobby/arcade-server functionality: a game/match/queue framework, per-player profiles, and a SQLite-backed user data store. It is split into two Gradle subprojects:
- **ArcadeAPI** — a `java`-only module of public interfaces/events other plugins depend on to integrate with ArcadeCore (package `org.drappula.arcadeApi`). Contains no implementation.
- **ArcadePlugin** — the actual plugin (`java-library` + Kotlin), package `org.drappula.arcadeCore`, implements every `ArcadeAPI` interface and is what gets shaded into the runnable jar.

## Build & run commands
Run from the `ArcadeCore/` root (this is the Gradle root project; `ArcadeAPI`/`ArcadePlugin` are subprojects).
- `./gradlew build` — compiles both modules and produces the shadowed plugin jar (shadowJar runs as part of `build`).
- `./gradlew :ArcadePlugin:shadowJar` — build the relocated, dependency-bundled jar directly (output in `ArcadePlugin/build/libs`).
- `./gradlew :ArcadePlugin:runServer` — launches a local Paper 1.21.11 test server with the plugin installed (via `xyz.jpenilla.run-paper`); use this for manual in-game testing. Server heap is set to `-Xms2G -Xmx2G`.
- There is no test suite configured in this project.

## Architecture
- **Language/toolchain**: Java 21 (ArcadeAPI) and Java 21 + Kotlin (ArcadePlugin, `kotlin("jvm")` applied, JVM toolchain 21), built with Gradle Kotlin DSL.
- **Shadow/relocation**: `com.gradleup.shadow` (applied only in `ArcadePlugin`) bundles `dev.dejvokep:boosted-yaml` (core + spigot-serializer) and `org.xerial:sqlite-jdbc`. `boosted-yaml` and `sqlite-jdbc` are *also* declared as runtime libraries in `ArcadeCoreLoader` (via a `MavenLibraryResolver` from Maven Central), which resolves them at plugin load time before the classloader is set up — the shadow-bundled copies and the loader-resolved copies coexist by design; don't assume removing one is safe without checking the other.
- **Plugin entry point**: `ArcadeCore` (`ArcadePlugin/src/main/java/org/drappula/arcadeCore/ArcadeCore.java`) is a singleton accessible via `ArcadeCore.get()`. `onEnable()` runs, in order: `setupConfig()` (loads `DataConfig`, `MainConfig`, `MessagesConfig`), `connectDatabase()` (opens the SQLite connection, throws on failure), `registerCommands()` (Brigadier via `LifecycleEvents.COMMANDS` — no `plugin.yml`, only `paper-plugin.yml`), `registerListeners()`, `registerAPI()` (publishes `ArcadeAPIImpl` through `ArcadeAPIProvider`). `onDisable()` reloads `GameManager` (clears in-memory game/match/participant state) and disconnects the database.
- **API/impl split**: Every cross-cutting system is defined as an interface in `ArcadeAPI` (`IGameManager`, `IMatchManager`, `IQueueManager`, `IMatch`, `IParticipant`) and implemented as a singleton (`X.get()` static accessor, lazily constructed) in `ArcadePlugin`'s `managers/` package (`GameManager`, `MatchManager`, `QueueManager`, `Match`, `Participant`). `ArcadeAPIImpl` (`ArcadePlugin/.../api/ArcadeAPIImpl.java`) is the single class that bridges the two, registered once in `ArcadeCore#onEnable`. When adding a new cross-plugin-facing feature, add the interface/event to `ArcadeAPI` first, then wire the concrete manager and expose it through `ArcadeAPIImpl`.
- **Game/match/queue flow**: `GameManager` holds registered `Game`s plus in-memory `matches`/`participants` maps keyed by game ID (no persistence — cleared on `reload()`, called from `onDisable`). `QueueManager.joinQueue` fires a cancellable `QueueEnterEvent`, and once a queue reaches `game.getPlayersRequired()` it slices off that many players and calls `MatchManager.startMatch`. `startMatch` fires a cancellable `MatchStartEvent`, populates the match into `GameManager`, and starts `MatchStartTask` (a repeating `BukkitRunnable`, countdown from `match.start-countdown` in `MainConfig`, ticks every 20 ticks/1s). `Match.end()` → `MatchManager.endMatch` fires `MatchEndEvent` and starts `MatchEndTask`, which — after a fixed 5-tick delay — eliminates all remaining participants (via `MatchManager.eliminateParticipant`, which applies `GameEndSettings` gamemode/fly overrides or config defaults) and then depopulates the match and sends spectators back to the lobby. There's no separate "declare winner" step wired up yet; `getWinnerParticipants()`/`setWinnerParticipants()` exist on `Match` but nothing currently calls the setter.
- **Config layer**: Config classes extend `Config` (`config/Config.java`), which wraps `dev.dejvokep.boostedyaml.YamlDocument` with `SpigotSerializer` (so Bukkit types like `Location` (de)serialize directly) and `BasicVersioning` keyed on a `version` field for auto-updating the file to the packaged default. Three configs, each with a static `setup()`/`get()`:
  - `DataConfig` (`data.yml`) — runtime/mutable state (currently just `SPAWN_LOCATION`, via `setSpawnLocation`/`getSpawnLocation`).
  - `MainConfig` (`config.yml`) — admin-facing settings, e.g. `lobby.teleport-spawn`, `lobby.gamemode`, `match.start-countdown`, `match.end.gamemode`, `match.end.fly-enabled`.
  - `MessagesConfig` (`messages.yml`) — all player-facing text (e.g. `match-started`, `countdown-message-singular`/`-plural`, `participant-eliminated`, `match-ended`, `bad-arguments.unknown-game`), sent via `MessageUtil.sendMessage` (MiniMessage + Adventure `Placeholder`/`TagResolver`).
- **Commands**: `MainCommand.get()` builds a single Brigadier tree under `/arcade`: bare `/arcade` prints plugin name/version/description; `reload` reloads `DataConfig` only (not `MainConfig`/`MessagesConfig`); `setspawn` saves the invoker's location; `start <game>` (requires `arcade.force-start` permission) force-starts a match via `QueueManager.forceStart`; `queue <game>` has tab-completion wired (`suggestGames`) but its handler is currently a no-op — it doesn't call `QueueManager.joinQueue`.
- **Database**: `Database` (`database/Database.java`) opens a single static SQLite connection to `<plugin-data-folder>/database.db` on enable and creates three tables if absent: `user_profiles` (uuid, username), `games` (game_id), `game_stats` (uuid, game_id, points, wins, losses, other_stats JSON — FKs cascade-delete). Only `user_profiles` is currently read/written, via `UserDataManager` (`get`/`getOrCreate`/`save`, upsert on conflict) — `games`/`game_stats` are provisioned but unused so far.
- **Listeners**: `LobbyListener` handles `PlayerJoinEvent` — teleports to `DataConfig`'s saved spawn (if `lobby.teleport-spawn` is enabled), resets health to max, sets gamemode from `MainConfig`.
- **Profiles**: `ProfileManager` is a simple in-memory `UUID -> Profile` map (register/get/remove) separate from the persisted `UserData`/`user_profiles` table — it's runtime session state, not database-backed.
- **Resource filtering**: `paper-plugin.yml` uses Gradle `processResources` token expansion for `${description}`/`${version}`, sourced from `gradle.properties`.
